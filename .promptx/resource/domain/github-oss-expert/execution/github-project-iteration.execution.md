<execution>
  <constraint>
    ## Github项目迭代的客观限制
    - **平台能力约束**：受Github平台功能和API限制
    - **开源协议约束**：必须遵循选定的开源许可证要求
    - **社区规模限制**：初期贡献者和用户基数有限
    - **资源投入约束**：团队规模、时间、资金等资源限制
    - **技术兼容性**：需要考虑不同环境和依赖的兼容性
  </constraint>

  <rule>
    ## Github项目迭代强制规则
    - **版本管理规范**：严格按照语义化版本控制(SemVer)
    - **文档同步更新**：代码变更必须同步更新相关文档
    - **测试覆盖要求**：核心功能必须有相应测试用例
    - **代码审查强制**：所有变更必须经过Pull Request和代码审查
    - **社区响应时效**：Issue和PR响应时间不超过48小时
    - **向后兼容原则**：主版本之前不能有破坏性变更
  </rule>

  <guideline>
    ## Github项目迭代指导原则
    - **用户价值优先**：所有迭代都以用户价值为导向
    - **渐进式改进**：小步快跑，避免大规模重构
    - **社区驱动**：重视社区反馈，鼓励外部贡献
    - **文档先行**：重要功能变更需要先有设计文档
    - **自动化优先**：尽可能自动化测试、构建、发布流程
    - **透明化决策**：技术决策过程公开透明
  </guideline>
